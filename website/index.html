Defusing logic bombs in symbolic execution with LLM-generated ghost code.

Gordian keeps an SMT solver in charge of global reasoning, then selectively gives it lightweight, LLM-generated aids where ordinary symbolic execution gets stuck.

Dimitrios Stamatios Bouras · Sergey Mechtaev · Peking University

Read on arXiv ↗

The Gordian idea

Do not replace precise symbolic reasoning. Generate solver aids only for the fragment that blocks it, then check every resulting input on the original program.

Engine
KLEE + SMT
Role of LLM
Targeted code generation
Safety
Original-program validation

Abstract

Symbolic execution is limited by solver-hostile program fragments, complex numerical reasoning, and unbounded heap structures. Gordian is a hybrid framework that uses LLMs selectively to generate lightweight ghost code that helps an SMT solver handle these fragments while preserving its precise, global reasoning capability.

Approach

Give the solver a way through, not a replacement.

Instead of asking an LLM to reason across an entire execution path, Gordian asks it to write a small piece of code with one purpose: make a particularly difficult fragment tractable for symbolic execution.

Three categories of ghost code

Each category changes where symbolic execution spends its effort. The generated code is deliberately narrow, and every resulting test is still validated on the unmodified program.

01

Ghost modeling

Rewrite an intractable fragment as a solver-friendly surrogate that preserves the branch behaviour needed by the suffix.

It generates
A small surrogate for one difficult fragment
It preserves
The control-relevant behaviour of the explored path
It is checked by
Replaying every candidate on the original program
02

Bidirectional propagation

Invert a difficult function so information from a desired suffix can move back toward the symbolic inputs.

It generates
An inverse or local search routine for the hard operation
It moves
Concrete suffix information back to symbolic inputs
It repeats
Prefix and suffix refinement until constraints agree
03

Semantic heap shapes

Partition an unbounded heap into a small set of domain-relevant topologies, then let execution explore each one symbolically.

It generates
Ghost builders and a symbolic selector over useful shapes
It partitions
Unbounded heaps into finite semantic topologies
It retains
Symbolic values and all unconstrained details

Three worked examples

Each block starts with the program obstacle, shows the targeted ghost-code aid, and ends at the same boundary: a candidate counts only after the original program accepts it.

Evidence

Tested where symbolic execution usually breaks down.

The evaluation combines synthetic logic bombs with numerical code and structured-input programs—three settings that stress different limits of symbolic execution.

Across benchmarks

Gordian improves coverage by 28.5–115.2% over the strongest traditional symbolic-execution baseline and by 74.1–189.8% over relevant LLM-based baselines, while reducing LLM token use by 91–96%.
